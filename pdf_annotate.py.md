# pdf_annotate.py

## 概要

セキュリティ監査用に生成した証跡PDFに対して、Excelの要件一覧に基づき、該当テキストのハイライト、項番ラベル、日本語翻訳を自動的に追記するツールです。

追記される全てのアノテーション（ハイライト・項番・翻訳）はPDFの編集可能オブジェクトとして作成されるため、Adobe Acrobat、Foxit Reader、PDF-XChange Editorなどのビューアで後から位置の移動や内容の編集が可能です。

さらに、アノテーションを追記したページとその前後1ページのみを抽出した軽量版PDFも自動生成します。

## 背景と目的

url2pdf.py で生成した証跡PDFは、Webページ全体をそのまま保存したものです。監査レビューの際には、PDF内のどの箇所がどの要件の証拠なのかを人間が特定する必要があり、数十〜数百ページに渡るPDFから該当箇所を探す作業は非常に手間がかかります。

本ツールはこの作業を自動化します。Excelに記載された証跡テキストをPDF内から検索し、該当箇所に黄色ハイライト、要件の項番（赤字）、日本語翻訳（青字）を自動で追記します。レビュー担当者はPDFを開くだけで「どこが」「何の要件の」証拠なのかを即座に把握できます。

## ワークフロー上の位置づけ

本ツールは以下のワークフローのステップ5に該当します。

| ステップ | 内容 | ツール |
|---|---|---|
| 1 | 人間がA〜D列（要件定義）を記入 | 手作業 |
| 2 | AIエージェントがE, F, G, H列を一括で埋める | AIエージェント |
| 3 | URL→ファイル名割り当て | assign\_filenames.py |
| 4 | URL→PDF生成 | url2pdf.py |
| **5** | **PDFにハイライト・翻訳を追記＋ページ抽出** | **pdf\_annotate.py（本スクリプト）** |

## 追記されるアノテーション

各証跡テキストに対して、以下の3種類のアノテーションが追記されます。

**黄色ハイライト：** G列の証跡テキストに一致するPDF内の箇所に付与されます。全ての一致箇所に対して常に付与されます。

**赤字の項番ラベル：** C列の項番（例: `2.20.8.1.`）が、ハイライトの左上付近にFreeTextアノテーションとして配置されます。同一ページ内で同じ項番が複数回出現する場合、ラベルは1回だけ描画されます（重複排除）。フォントサイズは7ptです。

**青字の日本語翻訳：** H列の日本語訳がハイライトの直下にFreeTextアノテーションとして配置されます。同一項番であっても各ハイライトごとに必ず描画されます（重複排除なし）。フォントサイズは6ptです。

全てのアノテーションはFreeText形式で作成されるため、PDFビューアで選択・移動・編集・削除が可能です。自動配置では位置がずれることもあるため、人間が後から微調整できる設計になっています。

## Excel列構成

本スクリプトが参照する列は以下の通りです。

| 列 | 列番号 | 内容 | 本スクリプトでの扱い |
|---|---|---|---|
| A | 1 | 大分類 | 参照しない |
| B | 2 | 中分類 | 参照しない |
| C | 3 | 項番 | 赤字ラベルのテキストとして使用 |
| D | 4 | 要件 | 参照しない |
| E | 5 | 回答 | 参照しない |
| F | 6 | URL | 参照しない |
| G | 7 | 証跡テキスト原文 | PDF内検索キーとして使用 |
| H | 8 | 日本語訳 | 青字翻訳テキストとして使用 |
| I | 9 | ページ番号 | 参照しない |
| J | 10 | 備考 | 参照しない |

### G列・H列のフォーマット

G列とH列は `[ファイル名|テキスト]` 形式で記述されている必要があります。

```
[EVD001|By default, the system has four user roles with a defined set of permissions.]
```

1つのセルに複数のエントリを含める場合は、ブラケットを続けて記述します。

```
[EVD001|テキスト1][EVD002|テキスト2]
```

ファイル名部分は、`--input-dir` に格納されているPDFのファイル名（拡張子なし）と一致する必要があります。テキスト部分は、そのPDF内に実際に含まれている文字列と一致する必要があります。

## 動作の仕組み

### 全体フロー

| 順序 | 処理 | 説明 |
|---|---|---|
| 1 | テキストインデックス構築 | 入力PDFの全ページからテキストを抽出しインデックス化 |
| 2 | Excel読み込み | G列・H列から証跡エントリをパースし、タスクを生成 |
| 3 | 重複排除 | 同一PDF・同一テキストの重複エントリを除外 |
| 4 | 既存チェック | 出力先に処理済みPDFがあればスキップ |
| 5 | テキスト検索 | インデックスを使って該当ページを特定 |
| 6 | アノテーション追記 | ハイライト・項番・翻訳をPDFに追加 |
| 7 | フォールバック検索 | 指定PDFで見つからない場合、他のPDFを検索 |
| 8 | ページ抽出 | コメント付きページ±1ページの軽量版PDFを生成 |
| 9 | サマリレポート | Excel行単位の処理結果を一覧表示 |

### テキストインデックスとキャッシュ

処理の冒頭で、入力ディレクトリ内の全PDFからページごとのテキストを抽出し、正規化（小文字化・空白統一）した上でインデックスを構築します。このインデックスにより、あるテキストがどのPDFの何ページ目にあるかを高速に検索できます。

インデックスはJSON形式でキャッシュされます（デフォルト: `--input-dir/.text_index_cache.json`）。各PDFの更新日時とファイルサイズをフィンガープリントとして記録し、変更がないPDFはキャッシュから読み込みます。PDFが追加・更新された場合のみ再インデックスが行われるため、2回目以降の実行は高速です。

### テキスト検索（段階的マッチング）

G列のテキストをPDF内から検索する際、完全一致で見つからない場合は段階的に検索範囲を緩和します。

| 段階 | 方法 | 説明 |
|---|---|---|
| 1 | 完全一致 | 正規化したテキスト全体でインデックスを検索 |
| 2 | 先頭部分一致（80%） | テキストの先頭80%で検索 |
| 3 | 先頭部分一致（60%） | テキストの先頭60%で検索 |
| 4 | 先頭部分一致（40%） | テキストの先頭40%で検索 |
| 5 | 先頭30文字 | テキストの先頭30文字で検索 |
| 6 | 先頭15文字 | テキストの先頭15文字で検索 |

この段階的マッチングにより、PDF化時に微妙な空白や改行が入った場合でもテキストを発見できます。

ページ特定後のハイライト追記時にも、`page.search_for` で同様の段階的な部分文字列検索を行います（80文字→50文字→30文字→20文字の順で試行）。

### フォールバック検索

指定されたPDF内にテキストが見つからない場合、入力ディレクトリ内の他の全PDFを検索します。たとえばG列に `[EVD001|テキスト]` と記載されているが、実際にはEVD002.pdfにそのテキストが存在する場合、EVD002.pdfにアノテーションが追記されます。サマリレポートではこのケースが「⚠FB」として表示されます。

### 重複排除

2つのレベルで重複排除を行います。

**Excelレベルの重複排除：** 同一PDF・同一テキストの組み合わせが複数のExcel行に存在する場合、アノテーション（ハイライト・ラベル・翻訳）は最初の行に対してのみ追記されます。後続の行にはサマリレポート上で「※重複」として処理結果が共有されます。

**ページレベルの項番重複排除：** 同一ページ内で同じ項番が複数のハイライトに紐づく場合、赤字の項番ラベルは1回だけ描画されます。日本語翻訳は各ハイライトごとに必ず描画されます。

### アノテーションの重なり回避

複数のアノテーションが同一ページに配置される場合、既に配置済みの矩形との衝突を検出し、衝突する場合は下方向にシフトして空き領域を探します（最大30回試行）。これにより、ラベルや翻訳が重なって読めなくなる問題を軽減しています。

### ページ座標系の正規化

一部のPDFは内部的に座標系が反転・回転していることがあり、テキスト追記が正しい位置に表示されない場合があります。本ツールは各ページの処理前に `page.clean_contents(sanitize=False)` を呼び出してページのコンテンツストリームを正規化し、座標系の問題を解決しています。`sanitize=False` を指定することで、既存のページコンテンツが削除される問題（PyMuPDF issue #4130）を回避しています。

### ページ抽出（軽量版PDF生成）

アノテーション追記後、コメントが追記されたページとその前後1ページのみを含む軽量版PDFを自動生成します。

ファイル名は `{元のファイル名}_抽出.pdf` です。たとえばEVD001.pdfの3ページ目と7ページ目にコメントがある場合、2・3・4・6・7・8ページ目を含む `EVD001_抽出.pdf` が生成されます。

ファイルサイズを抑えるため、元PDFをそのまま開いて不要ページを削除する方式を採用しています（新規PDFにページをコピーする方式ではフォントや画像リソースが重複し、元より大きくなる問題がありました）。保存時に `garbage=4`（未使用オブジェクトの完全除去）と `deflate=True`（圧縮）を適用します。

### 既存ファイルのスキップ

出力ディレクトリにアノテーション済みPDF（`{stem}.pdf`）が既に存在する場合、そのPDFへのアノテーション処理をスキップします。サマリレポート用の集計は行います。抽出PDF（`{stem}_抽出.pdf`）が既に存在する場合も同様にスキップします。再実行しても既存の成果物を壊さない設計です。

### サマリレポート

処理完了後、Excel行（要件）単位でどのPDFの何ページ目にマーキングしたかを一覧表示します。

```
============================================================
  サマリレポート（Excel要件単位）
============================================================
    行  項番            結果  マーキング箇所
------------------------------------------------------------
     2  1.1.1            ✓   EVD001.pdf(3p), EVD001.pdf(5p)
     3  1.1.2            ✓   EVD002.pdf(1p)
     4  1.2.1            ⚠   EVD003.pdf(2p)⚠FB
     5  1.2.2            △   EVD004.pdf(4p), EVD005.pdf ✗未検出
     6  1.3.1            ✗   EVD006.pdf ✗未検出
    ...
------------------------------------------------------------
  合計: ✓成功 120件  ⚠FB 8件  ✗未検出 12件
============================================================

  凡例: ✓=全て成功  ⚠=フォールバック先で発見  △=一部未検出  ✗=全て未検出
        ※重複=同一テキストが別行で既にマーキング済み（ハイライトは共有）
```

各行のステータスは以下の意味を持ちます。

| ステータス | 意味 |
|---|---|
| ✓ | その行の全証跡が指定PDFで見つかり、マーキング成功 |
| ⚠ | 指定PDFでは見つからなかったが、別のPDFで見つかった（フォールバック） |
| △ | 一部の証跡は成功したが、一部は見つからなかった |
| ✗ | 全ての証跡が見つからなかった |
| ※重複 | 同一テキストが別の行で既にマーキング済み |

## 前提条件と環境構築

### 必要な環境

本スクリプトの実行には以下が必要です。

Python 3.8以上がインストールされていること。`python3 --version` で確認できます。

入力PDFが格納されたディレクトリ（デフォルト `./pdf`）が存在し、url2pdf.py による PDF 生成（ワークフローのステップ4）が完了していること。

Excelファイルのステップ1〜3（A〜D列の記入、AIによるE〜H列の記入、assign_filenames.py によるファイル名割り当て）が完了していること。

### Python パッケージのインストール

仮想環境の使用を推奨します。

```bash
# 仮想環境の作成と有効化（初回のみ）
python3 -m venv url2pdf-env
source url2pdf-env/bin/activate    # Linux / macOS
# url2pdf-env\Scripts\activate     # Windows

# 必要パッケージのインストール
pip install openpyxl pymupdf
```

各パッケージの役割は以下の通りです。

`openpyxl` はExcelファイル（.xlsx）の読み込みに使用します。`pymupdf` はPDFの読み込み、テキスト抽出、アノテーション追記、ページ削除に使用します。PyMuPDF 1.24.0以降が必要です（`import pymupdf` 形式でのインポートに対応したバージョン）。

### インストールの確認

```bash
python -c "import openpyxl; print('openpyxl:', openpyxl.__version__)"
python -c "import pymupdf; print('pymupdf:', pymupdf.__version__)"
```

両方のバージョンが表示されれば準備完了です。

### ディレクトリ構成の確認

実行前に以下の構成になっていることを確認してください。

```
作業ディレクトリ/
├── pdf_annotate.py          ← 本スクリプト
├── input.xlsx               ← ステップ3完了済みのExcel
└── pdf/                     ← ステップ4で生成されたPDF群
    ├── ZBX001.pdf
    ├── ZBX002.pdf
    ├── VEM001.pdf
    └── ...
```

出力ディレクトリ（デフォルト `./pdf_annotated`）は実行時に自動作成されます。

## 使い方

### 基本

```bash
python pdf_annotate.py input.xlsx --input-dir=./pdf --output-dir=./pdf_annotated
```

### オプション指定

```bash
# キャッシュファイルを明示
python pdf_annotate.py input.xlsx \
    --input-dir=./pdf \
    --output-dir=./pdf_annotated \
    --cache=./my_cache.json
```

### 再実行

出力ディレクトリに既存ファイルがある場合は自動スキップされるため、同じコマンドを再実行しても安全です。全て再処理したい場合は、出力ディレクトリを削除してから実行してください。

```bash
rm -rf ./pdf_annotated
python pdf_annotate.py input.xlsx --input-dir=./pdf --output-dir=./pdf_annotated
```

| オプション | デフォルト | 説明 |

## 出力ファイル

| ファイル名 | 内容 |
|---|---|
| `{stem}.pdf` | アノテーション追記済みPDF（全ページ） |

アノテーション対象でないPDFも、入力ディレクトリから出力ディレクトリにそのままコピーされます。

## 注意事項

- G列のテキストはPDF内のテキストと正確に一致する必要があります。AIエージェントが要約や言い換えを行うと、パターンマッチに失敗しマーキングが行われません。
- 本スクリプトはExcelの2行目からデータが存在する最終行までを自動的に処理します。
- `page.clean_contents(sanitize=False)` による座標系正規化は大半のPDFで有効ですが、特殊な構造のPDFでは効果がない場合があります。
