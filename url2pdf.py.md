# url2pdf.py

## 概要

セキュリティ監査や要件適合性チェックにおいて、Webページを証跡（エビデンス）としてPDF保存するためのツールです。

Excelファイルに記載されたURLリストを読み取り、各URLのWebページをヘッドレスブラウザ（Chromium）で開き、ページ全体をPDFとして保存します。複数ワーカーによる並列処理に対応しており、大量のURLを効率的に処理できます。

## 背景と目的

セキュリティ監査では「この製品はこの要件を満たしている」ことを証明するために、製品ドキュメントのWebページを証跡として保存する必要があります。しかし手作業でブラウザからPDF保存すると、以下の問題が発生します。

- URLが数十〜数百件になると作業時間が膨大になる
- Cookieバナーが表示されたままPDF化される
- 遅延読み込み画像が表示されない
- SPA（Single Page Application）でページ全体が印刷されない
- ヘッダー・フッターに取得日時やURLが記録されない

本ツールはこれらの問題を自動的に解決し、監査に適した高品質なPDFを一括生成します。

## ワークフロー上の位置づけ

本ツールは以下のワークフローのステップ4に該当します。

| ステップ | 内容 | ツール |
|---|---|---|
| 1 | 人間がA〜D列（要件定義）を記入 | 手作業 |
| 2 | AIエージェントがE, F, G, H列を一括で埋める | AIエージェント |
| 3 | URL→ファイル名割り当て（I列に書き込み） | assign\_filenames.py |
| **4** | **URL→PDF生成（F列のURL、I列のファイル名を使用）** | **url2pdf.py（本スクリプト）** |
| 5 | PDFにハイライト・翻訳を追記＋索引Excel出力（J列） | pdf\_annotate.py |

## 動作の仕組み

### 全体フロー

| 順序 | 処理 | 説明 |
|---|---|---|
| 1 | Excel読み込み | F列（URL）とI列（ファイル名）を読み取り、ペアを生成 |
| 2 | 重複排除 | 同一ファイル名のタスクを除外（同じURLから2度生成しない） |
| 3 | 既存チェック | 出力先に正常サイズのPDFが既にあればスキップ |
| 4 | ワーカー起動 | 指定数のワーカープロセスを起動 |
| 5 | PDF生成 | 各ワーカーがChromiumでページを開きPDF化 |
| 6 | 結果集計 | 成功・失敗・スキップの件数とログを出力 |

### Cookieバナー除去（2段階）

現代のWebサイトの多くはGDPR対応のCookieバナーを表示します。これがPDFに含まれるとページの大部分が隠れてしまうため、2段階のアプローチで除去しています。

**第1段階：ボタンクリックによる正規の閉じ方**

主要なCookie同意プラットフォーム（OneTrust、Cookiebot等）のボタンセレクタと、一般的なボタンテキスト（"Accept"、"Accept All"、"同意する"など、英語13種・日本語4種の計17種）を順に試行し、表示されているボタンが見つかればクリックします。

**第2段階：CSS強制非表示**

ボタンクリックでは消えないバナーやオーバーレイに対応するため、Cookie関連のDOM要素（IDやクラス名に "cookie"、"consent"、"gdpr" 等を含む要素）をJavaScriptで一括非表示にします。さらに、バナーに伴うモーダルバックドロップやスクロールロックも解除します。

この2段階処理はPDF生成前に2回実行されます。初回はページ読み込み直後、2回目はスクロール・レイアウト修正後です。スクロール操作によってバナーが再表示されるケースに対応しています。

### 遅延読み込み画像への対策

多くのWebサイトはパフォーマンス最適化のため、画像の遅延読み込み（lazy loading）を採用しています。ヘッドレスブラウザでは画面外の画像が読み込まれないため、以下の対策を実施しています。

まず、`loading="lazy"` 属性を持つimg要素を `loading="eager"` に変更し、即座に読み込みを開始させます。次に、`data-src` 属性に実際の画像URLが格納されている要素（JavaScriptライブラリによる遅延読み込みパターン）について、`src` がプレースホルダーやdata URIの場合は `data-src` の値を `src` にコピーします。

その後、ページ全体をビューポート高さの80%刻みでスクロールし、各ステップで400msの待機を入れます。これによりスクロール連動型の遅延読み込みが発動し、ページ内の全画像が読み込まれます。最後にページ先頭に戻ります。

### SPA・Docusaurusレイアウト修正

SPA（Single Page Application）やDocusaurusベースのドキュメントサイトでは、CSSの `overflow: hidden` や `height: 100vh` などの指定により、PDF化時にページの一部しか印刷されない問題が頻繁に発生します。

本ツールはページ内の全DOM要素を走査し、以下の修正を適用します。

- `overflow: hidden` / `overflow-y: hidden` → `visible` に変更
- ビューポート単位（vh）の高さ指定 → `auto` に変更
- `max-height` の制限 → 解除（`none`）
- `position: sticky` → `relative` に変更
- `position: fixed` → `absolute` に変更

さらにDocusaurus固有の対策として、`#__docusaurus` コンテナや `docPage`、`docMainContainer`、`docRoot` などのクラスを持つ要素に対して、高さとオーバーフローの制限を解除します。

### PDF出力時の画像読み込み完了待機

PDF生成の直前に、まだ読み込みが完了していない画像があれば、各画像の `onload` / `onerror` イベントを待機します。個々の画像に対して最大5秒のタイムアウトを設定し、画像の読み込みが完了するかタイムアウトするかを待ちます。

### ヘッダー・フッター

生成されるPDFには自動的にヘッダーとフッターが付与されます。

**ヘッダー（上部）：**

| 左側 | 右側 |
|---|---|
| ページタイトル（最大80文字） | URL |

下線付きで本文と視覚的に区切られます。

**フッター（下部）：**

| 左側 | 右側 |
|---|---|
| 取得日時（JST）・ファイル名 | ページ番号（現在/総ページ） |

上線付きです。これにより、PDFを見ただけで「いつ」「どのURLから」取得した証跡であるかが明確にわかります。

### 並列処理アーキテクチャ

処理速度を向上させるため、マルチプロセスによる並列処理を実装しています。

メインプロセスがタスクキューにURL/ファイル名ペアを投入し、指定数のワーカープロセスがキューからタスクを取り出して処理します。各ワーカーは独立したChromiumブラウザインスタンスを保持し、タスクを順次処理します。キューの末尾にはワーカー数分のセンチネル値（None）が投入され、ワーカーはこれを受信すると終了します。

デフォルトのワーカー数は5です。URLの数がワーカー数より少ない場合は、URL数に合わせてワーカー数が自動的に縮小されます。

### リトライ機構

ネットワークエラーやタイムアウトが発生した場合、指定回数（デフォルト1回）まで自動リトライします。リトライ間隔は3秒です。ページ読み込みはまず `networkidle`（ネットワーク通信が完全に停止するまで待機）を試み、タイムアウトした場合は `domcontentloaded`（DOM構築完了）にフォールバックします。

### 既存ファイルのスキップ

出力先ディレクトリに同名のPDFが既に存在し、そのファイルサイズが閾値（デフォルト60KB）以上であれば、正常に生成済みと判断してスキップします。閾値未満のファイルは不完全と判断して再ダウンロード対象になります。`--force` オプションを指定すると、既存ファイルに関係なく全てのURLを再処理します。

## 前提条件と環境構築

### 必要な環境

本スクリプトの実行には以下が必要です。

Python 3.8以上がインストールされていること。`python3 --version` で確認できます。

Excelファイルのステップ1〜3が完了していること。具体的には、人間によるA〜D列の記入、AIエージェントによるE〜H列の記入、`assign_filenames.py` によるファイル名割り当て（F列のURL→I列のファイル名、G列・H列のURL置換）が済んでいる必要があります。特にF列（URL）とI列（通し番号ファイル名）が正しく対応していることが前提です。

### Python パッケージのインストール

仮想環境の使用を推奨します。

```bash
# 仮想環境の作成と有効化（初回のみ）
python3 -m venv url2pdf-env
source url2pdf-env/bin/activate    # Linux / macOS
# url2pdf-env\Scripts\activate     # Windows

# 必要パッケージのインストール
pip install openpyxl playwright
```

`openpyxl` はExcelファイル（.xlsx）の読み込みに使用します。`playwright` はヘッドレスChromiumブラウザの制御に使用します。それ以外の依存パッケージはすべてPython標準ライブラリ（`argparse`、`re`、`sys`、`os`、`time`、`datetime`、`pathlib`、`multiprocessing`）のため、追加インストールは不要です。

### Chromium ブラウザのインストール

Playwrightが内部的に使用するChromiumブラウザを別途インストールする必要があります。これはPlaywrightが管理する専用のブラウザであり、システムにインストール済みのChromeやChromiumとは独立しています。

```bash
playwright install chromium
```

初回実行時に約150〜200MBのダウンロードが発生します。ネットワーク接続が必要です。このコマンドは仮想環境を有効化した状態で実行してください。

### インストールの確認

```bash
python -c "import openpyxl; print('openpyxl:', openpyxl.__version__)"
python -c "from playwright.sync_api import sync_playwright; print('playwright: OK')"
playwright install --dry-run chromium 2>&1 | head -3
```

openpyxlのバージョン表示、playwrightのインポート成功、Chromiumのインストール状態が確認できれば準備完了です。

### 入力ファイルの確認

実行前に、Excelファイルとディレクトリが以下の状態になっていることを確認してください。

```
作業ディレクトリ/
├── url2pdf.py                          ← 本スクリプト
├── input_named_YYYYMMDD_HHMMSS.xlsx    ← ステップ3完了済みのExcel
│                                          F列: URL が記入済み
│                                          I列: ファイル名が記入済み
│                                                （assign_filenames.py が生成）
└── pdf/                                ← 出力先（自動作成される）
```

F列に複数のURLがある行では、I列にも同じ数のファイル名が改行区切りで記載されている必要があります。この対応関係は `assign_filenames.py` によって自動的に設定されます。

## 使い方

### 基本

```bash
python url2pdf.py input.xlsx -o ./pdf
```

### 全オプション指定

```bash
python url2pdf.py input.xlsx \
    -o ./pdf \
    -w 5 \
    -f A4 \
    -l \
    -r 2 \
    -j 3 \
    --min-size 60000 \
    --force
```

### よくある使い方

```bash
# A4縦で5ワーカー（デフォルト）
python url2pdf.py input.xlsx -o ./pdf

# 横長のページが多い場合
python url2pdf.py input.xlsx -o ./pdf -l

# ネットワークが不安定な場合（リトライ3回、待機10秒）
python url2pdf.py input.xlsx -o ./pdf -r 3 -w 10

# 全て再取得
python url2pdf.py input.xlsx -o ./pdf --force

# ワーカー数を抑えてCPU負荷を軽減
python url2pdf.py input.xlsx -o ./pdf -j 2
```

## コマンドラインオプション一覧

| オプション | デフォルト | 説明 |
|---|---|---|
| `excel`（必須） | – | 入力Excelファイル |
| `-o, --output-dir` | `.`（カレントディレクトリ） | PDF出力ディレクトリ |
| `-w, --wait` | `5` | ページ読み込み後の追加待機秒数 |
| `-f, --format` | `A4` | 用紙サイズ（A4, Letter, Legal等） |
| `-l, --landscape` | off | 横向き印刷 |
| `-r, --retry` | `1` | 失敗時のリトライ回数 |
| `-j, --workers` | `5` | 並列ワーカー数 |
| `--min-size` | `60000` | 正常PDFとみなす最小バイト数（これ未満は再取得） |
| `--force` | off | 既存PDFを強制上書き |

## Excel列構成

本スクリプトが参照する列は以下の通りです。

| 列 | 列番号 | 内容 | 本スクリプトでの扱い |
|---|---|---|---|
| F | 6 | 証跡URL | URLを抽出 |
| I | 9 | ファイル名 | 出力PDFのファイル名（.pdfは自動付与） |

その他の列は本スクリプトでは参照しません。参考として、I列は `assign_filenames.py`（ステップ3）が書き込み、J列は `pdf_annotate.py`（ステップ5）が索引情報を書き込みます。

F列に複数のURLがある場合、I列にも対応する数のファイル名が改行区切りで記載されている必要があります。URLとファイル名は上から順に1対1で対応します。

## 生成されるPDFの特徴

生成されるPDFは以下の特徴を持ちます。

- ヘッダーにページタイトルとURL、フッターに取得日時・ファイル名・ページ番号が記載され、証跡としてのトレーサビリティが確保されます。
- Cookieバナーやオーバーレイが除去されており、ページ本文が完全に表示されます。
- 遅延読み込み画像が全て読み込まれた状態でPDF化されます。
- SPAやDocusaurusサイトでもページ全体が正しく印刷されます。
- 背景色や背景画像が保持されます（`print_background: true`）。

## 注意事項

Playwrightの初回実行時にChromiumのダウンロードが必要です（`playwright install chromium`）。

ヘッドレスブラウザを使用するため、GUI環境は不要です（サーバー上でも動作可能）。

JavaScript認証が必要なページ（ログインが必要なページ）には対応していません。

ワーカー数を増やすとメモリ使用量が増加します。各ワーカーが独立したChromiumインスタンスを起動するため、1ワーカーあたり約200〜500MBのメモリを消費します。

`--wait` の値は対象サイトの応答速度に応じて調整してください。画像やスクリプトの読み込みが遅いサイトでは値を大きくすると成功率が上がります。

出力PDFのファイルサイズが `--min-size` 未満の場合、再実行時に自動的に再取得されます。完全に空のPDFや壊れたPDFを検出する仕組みです。

本スクリプトはExcelの2行目からデータが存在する最終行までを自動的に処理します（`ws.max_row` を使用した動的検出）。
